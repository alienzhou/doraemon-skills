# 对话蒸馏器（Conversation Distiller）

一个元能力：把**一次性的思考**，升级成**可复利的资产**。

它作用的对象不是某个话题，而是**对话本身**。它在旁边观察一段讨论，判断：

> 这里有没有一个「下次还会用到」的东西？如果有，它该被固化成什么形态？

---

## 核心信念

大多数有价值的思考，是在对话里即兴发生的，说完就蒸发了。「知道」和「可复用」之间，隔着一次**主动的抽象**——而这次抽象几乎没人做。这个 Skill 就是那次抽象的执行者。

它捕捉的是这样的迁移：

```
散落在一次次对话里、说完就忘的隐性方法
        ↓
被实时识别、抽象成显性结构、固化成能反复调用的资产
```

---

## 与 skill-creator 的分工（务必分清，避免重叠）

- **conversation-distiller（本 Skill）**：管上游——**识别该沉淀什么 + 判断值不值得 + 决定用什么形态 + 抽象出结构骨架**。
- **skill-creator（平台基础设施）**：管下游——把已想清楚的东西**落地成 Skill 文件**，是创建 Skill 的唯一入口。

**硬规则**：当蒸馏形态判定为 Skill 时，本 Skill **不自己写 SKILL.md / manifest.json**，而是把「已抽象好的骨架」移交给 skill-creator 完成创建。本 Skill 只负责把「一团对话」变成「一份想清楚的骨架」。

---

## 触发方式：仅由明确的复用意图触发

只有用户明确要求把对话内容**抽象成可复用资产**时才进入蒸馏流程，例如：

- 「蒸馏一下这套方法」
- 「把刚才的判断框架固化下来」
- 「这套流程以后还要用，抽象一下」
- 「先整理成可复用骨架，再考虑做成 Skill」

对话即使自然形成了清晰框架，也不主动提示、不默认启动蒸馏。是否把一次讨论升级为长期复用资产，应由用户明确决定。

### 明确不触发的近似表达

以下表达默认只要求保存当前内容，不代表抽象或复用：

- 「收录进来」「记录一下」
- 「保存这段讨论」「写成记忆文档」
- 「把聊天整理成文档」「归档到仓库」

遇到这些请求，按当前仓库的普通文档规范直接归档，保留讨论对象、脉络和结论，不剥离具体上下文，也不自行改造成 Skill、手册或方法论。

只有当用户进一步明确提出「可复用」「抽象」「固化方法」「蒸馏」时，才使用本 Skill。

## 蒸馏流程

### 步骤 1 · 抽离：这段对话真正沉淀出了什么？

从第三视角复述：**剥掉具体话题，留下可迁移的结构**。
区分三层，指出哪一层值得留下：

```
对象层：我们聊的具体话题（通常不值得单独沉淀）
方法层：从中提炼出的可迁移方法/流程/框架（← 最常见的蒸馏对象）
元层：这次「怎么想/怎么做」的动作本身（更通用，杠杆更大）
```

### 步骤 2 · 判值：值不值得沉淀？

快速过一遍：
- **复现频率**：下次还会遇到吗？（不会 → 不蒸馏）
- **抽象增益**：显性化后比「每次现想」省多少？（几乎不省 → 不蒸馏）
- **通用度**：只对这一个项目有用，还是跨场景可用？
- **可固化度**：能被写成稳定结构吗，还是本质是即兴判断？

判定为「不值得」时，坦白告诉用户并说明理由，不硬做。

### 步骤 3 · 定形态：蒸馏成什么？（多形态路由）

根据「东西的性质」选载体，**首选 Skill**（通用标准格式），但不是唯一：

| 沉淀物性质 | 推荐形态 | 说明 |
| --- | --- | --- |
| 可反复触发、有稳定流程、希望以后一句话唤起 | **Skill**（首选） | 移交 skill-creator 落地 |
| 知识密集、需要查阅、步骤多但不需触发式调用 | 指导手册 / 文档 | 写成 markdown 文档 |
| 一组要点/约束/检查项 | 清单 / Checklist | 轻量，可内嵌进已有文档 |
| 一次判断的结论与理由 | 决策记录 / 卡片 | 沉淀「为什么这么定」 |
| 反复要填的结构 | 模板 | 供后续套用 |

一段对话可能同时产出多个形态（如：一个 Skill + 一份配套手册）。

### 步骤 4 · 抽骨架：把它整理成「想清楚的一份东西」

产出一份**可直接交付下游的骨架**，至少包含：

```
名称（给它一个准确的名字）：
一句话定位（它让谁，从什么状态，到什么状态）：
核心结构（步骤 / 框架 / 清单 / 决策树）：
触发/使用时机：
边界（不适用于什么）：
形态决策（skill / 手册 / 文档 / 模板，及理由）：
```

### 步骤 5 · 路由落地

- 形态 = **Skill** → 把上面的骨架移交 **skill-creator**，由它走正式创建流程（判重、写文件、质量校验）。本 Skill 到此完成。
- 形态 = **手册/文档/模板/清单** → 先读取当前仓库的目录规范，再按用户明确指定的位置落地。
- 用户没有指定专题项目时，不得根据对话主题自行推断 `projects/{主题}/`；使用仓库定义的通用文档位置。只有用户明确指定某个项目，或当前任务本来就在维护该项目时，才能写入对应项目目录。

---

## 递归自觉（这个 Skill 的一个特性）

这个元能力可以作用于自己：做事 → 沉淀「怎么做事」→ 再识别「怎么沉淀」也是一种能力……每上升一层，复用的杠杆更大。
使用时不必刻意追求递归，但当用户表现出「想抽离得更高一层」时，可以顺着帮他往上抽一层——这往往是最有价值的蒸馏。

---

## 收尾

蒸馏完成时，用一句话回给用户「这次留下了什么、放在哪、以后怎么再用」，让资产真正可被再次调用，而不是又变成一段说完就忘的对话。
